# 反馈驱动开发记录：修正 probe_412 探针以复现 412 问题

## 1. 背景

### 1.1 问题描述

现有的 `probe_412_test.go` 探针无法触发 B 站 412 风控，但实际 crawler 全量运行时每次都会触发 412。

**根因分析**：探针是在全新的 browser session 上"冷启动"测试，而实际 crawler 在进入 Stage 1 时已经经历了 Stage 0 的大量请求（24 页搜索，40 秒内），session 已处于"高风险"状态。探针缺少这个"请求累积"的前置条件。

### 1.2 验证目标

重写 probe_412 探针，使其模拟真实 crawler 的请求模式：
1. **Phase 1（模拟 Stage 0）**：用 `NavigateAndExtract` 连续访问多页搜索结果，模拟 Stage 0 的请求累积
2. **Phase 2（模拟 Stage 1）**：用 `NavigateAndIntercept` 访问博主视频页，模拟 Stage 1 的请求模式
3. **成功标准**：探针能在 Phase 2 中观察到 412 或 intercept timeout 错误

### 1.3 运行命令

```bash
go test -v -run TestProbe412_RealisticWorkload -timeout 600s ./src/platform/bilibili/ 2>&1 | tee /tmp/probe_412_fix.log
```

## 2. 迭代过程

### 2.1 迭代 1：串行 Phase 1 + 单步 Phase 2（v1）

**修改**：重写 `probe_412_test.go`，Phase 1 串行访问 15 页搜索，Phase 2 串行访问 10 个作者的 video 页。

**结果**：15/15 页 + 10/10 作者全部成功，0 个 412。

**分析**：串行请求密度不够，真实 crawler 的 Stage 0 是并发 3 个 worker。

### 2.2 迭代 2：并发 Phase 1 + 单步 Phase 2（v2）

**修改**：Phase 1 改为并发 3 个 worker，页数增加到 24（与真实 crawler 一致）。

**结果**：24/24 页 + 15/15 作者全部成功，0 个 412。

**分析**：Phase 2 只做了 `FetchAllAuthorVideos`（1 步），真实 crawler 每个作者做 2 步（info + video）。

### 2.3 迭代 3：并发 Phase 1 + 双步 Phase 2（v3）— 卡住

**修改**：Phase 2 增加 `FetchAuthorInfo` 步骤（先 info 再 video），作者数增加到 20。

**结果**：**测试卡住**。Phase 1 成功，Phase 2 在处理 `mid=3546983612549951` 时卡住。

**根因**：该用户没有视频，`/video` 页面不触发 `arc/search` API，`NavigateAndIntercept` 使用了探针的 8 分钟全局 ctx 而非 30s 的 `defaultInterceptTimeout`（因为 ctx 有 deadline，跳过了默认超时逻辑）。

**修复**：为每个作者创建独立的 30s 超时子 context。

### 2.4 迭代 4：修复超时 + 正常间隔（v4）

**修改**：每个作者使用 `context.WithTimeout(ctx, 30s)` 子 context。

**结果**：24/24 页 + 20 作者（19 OK, 1 timeout），0 个 412。timeout 是 `mid=3546983612549951`（无视频用户），不是风控。113 秒完成。

### 2.5 迭代 5：半间隔（v5）

**修改**：interval 减半为 1.25s。

**结果**：20/20 作者全部成功，0 个 412。64 秒完成。

### 2.6 迭代 6：零间隔（v6）

**修改**：interval 设为 0ms（极限压力）。

**结果**：20/20 作者全部成功，0 个 412。37 秒完成。

## 3. 结论

### 3.1 核心发现

**412 无法通过单次探针运行复现**。即使在极限条件下（零间隔、24 页搜索 + 20 作者双步请求 = 64 次导航，37 秒内完成），B 站也没有返回 412。

### 3.2 原因分析

B 站的 412 风控机制**不是简单的"请求频率"触发**，而是基于更复杂的因素：

1. **会话累积风险**：真实 crawler 触发 412 时，cookie/session 可能已经因为之前的多次测试运行被标记为"高风险"。探针每次运行虽然用同一个 cookie，但距离上次大量请求已过了数小时，风控计数器可能已重置。

2. **时间窗口效应**：真实 crawler 在凌晨 3:26 运行，探针在上午 9:45~9:51 运行。B 站可能在不同时段有不同的风控阈值。

3. **请求总量阈值**：真实 crawler 的 Stage 1 要处理 448 个作者（探针只有 20 个），虽然 412 在前 3 个作者就触发了，但可能是之前运行的累积效应。

4. **指纹/行为分析**：B 站可能基于浏览器指纹、请求模式的"机器人特征"等进行综合判断，而非单纯的频率限制。

### 3.3 探针的价值

虽然未能复现 412，但探针仍有重要价值：

1. **修复了 `NavigateAndIntercept` 的超时问题**：发现当 caller 传入有 deadline 的 ctx 时，`defaultInterceptTimeout` 不生效，导致无视频用户的请求会卡住很久。
2. **验证了当前配置的安全性**：在 2.5s 间隔 + 3 并发下，24 页搜索 + 20 个作者是安全的。
3. **建立了可复用的测试框架**：未来如果需要测试风控，可以快速调整参数。

### 3.4 后续建议

1. **在真实 crawler 中添加 412 计数器和日志**：记录每次运行的 412 触发时间、请求总量、距离上次运行的时间间隔，积累数据以理解风控规律。
2. **考虑在 Stage 间添加冷却时间**：即使探针未触发 412，真实 crawler 的 448 个作者仍然可能触发，建议在 Stage 0 和 Stage 1 之间添加 10-30s 的冷却。
3. **自适应限速**：遇到 412 时自动增加间隔，而非固定重试。
