## 1. 项目背景

### 1.1 项目定位

视频平台元数据爬虫工具，主要用于**网红-品牌营销**场景。

标杆产品：[NOX Influencer](https://cn.noxinfluencer.com/)。NOX 是一个完整的 SaaS 平台，具备网红数据分析、品牌数据分析、排行榜、AI 筛选网红等能力。我们的目标是循序渐进地实现类似能力。

### 1.2 已有实践

此前使用 Python + Selenium 实现了 YouTube 爬虫（`doc/youtube.py`），核心流程：
- 输入关键字 → Selenium 打开搜索页 → 滚动到底 → 解析 HTML 中的 `aria-label` 提取元数据 → 输出 CSV
- 提取字段：`["标题", "作者", "播放次数", "发布时间", "视频时长(s)", "来源"]`

### 1.3 痛点

- Python Scrapy 等静态爬虫无法处理 SPA/动态渲染页面
- 逆向获取后台 API 接口对 Instagram 等封闭平台不现实
- Python 部署依赖多，性能有限

## 2. 技术决策

### 2.1 语言选择：Go

| 考量 | 结论 |
|------|------|
| 性能 | Go 编译型语言，性能远优于 Python |
| 依赖管理 | Go Modules 简洁，不像 Python 需要一堆依赖，也不像 C++ 难以管理 |
| 部署 | 编译为单二进制，无需运行时环境 |
| 并发 | goroutine 天然适合多平台并行爬取 |
| 跨平台 | 原生交叉编译（`GOOS=windows/linux/darwin`），**优先 Windows 开发** |

### 2.2 爬虫策略：分梯度混合方案

单一技术方案无法覆盖所有平台，采用**按平台选择最优策略**的混合方案：

#### 三类爬虫技术对比

| 技术 | 代表库 | 适用场景 | 局限 |
|------|--------|---------|------|
| **静态 HTML 爬虫** | Colly, Goquery | 静态页面 | 无法处理 SPA/动态渲染，**不适用本项目** |
| **API 直调** | Go HTTP 客户端 | 有公开 API 的平台 | 依赖接口稳定性，封闭平台不可行 |
| **浏览器自动化** | chromedp, Rod, Playwright-go | 所有平台（兜底） | 速度慢、资源消耗高 |

#### 各平台推荐方案

| 平台 | 推荐方案 | 理由 |
|------|---------|------|
| **Bilibili** | API 直调（HTTP 客户端） | B站 API 文档齐全（[bilibili-API-collect](https://github.com/SocialSisterYi/bilibili-API-collect)），搜索接口支持分页，效率远高于浏览器模拟 |
| **YouTube** | YouTube Data API v3（官方 API） | 官方提供搜索和视频统计接口，有配额但够用；配额不够再降级到浏览器自动化 |
| **Instagram** | 浏览器自动化（Rod / chromedp） | API 封锁严格，只能走浏览器模拟登录 + 滚动抓取 |
| **其他未知平台** | 浏览器自动化作为兜底 | 通用性最强 |

#### Go 浏览器自动化库对比

| 库 | 原理 | 说明 |
|---|---|---|
| **chromedp** | Chrome DevTools Protocol (CDP) | Go 原生，直接通过 CDP 协议控制 Chrome/Chromium，无需额外驱动。Selenium 的最佳替代品 |
| **Rod** | Chrome DevTools Protocol (CDP) | 比 chromedp 更高层的封装，API 更友好，自动管理浏览器生命周期，支持自动下载 Chromium。**开发体验最好** |
| **Playwright for Go** | Playwright 协议 | 微软 Playwright 的 Go 绑定，支持 Chromium/Firefox/WebKit，反检测能力强 |
| **selenium (Go binding)** | WebDriver 协议 | Go 版 Selenium，社区活跃度低，**不推荐** |

### 2.3 跨平台风险评估

| 技术 | 跨平台情况 | 风险 |
|------|-----------|------|
| Go 本身 | ✅ 原生交叉编译 | 无风险 |
| HTTP 客户端（API 直调） | ✅ 纯 Go 实现 | 无风险 |
| Rod（浏览器自动化） | ✅ 自动下载对应平台的 Chromium | 低风险，需 headless 模式 |
| chromedp | ✅ 依赖系统安装的 Chrome/Chromium | 需要用户自行安装浏览器 |
| CSV 导出 | ✅ 纯 Go 标准库 | 无风险 |

**结论**：Go + Rod/chromedp 方案天然跨平台，没有实质性障碍。

### 2.4 架构方向

```
┌──────────────┐
│  用户输入     │  关键字 + 平台选择
└──────┬───────┘
       ▼
┌──────────────┐
│  Crawler 接口 │  统一的 Crawler interface
└──────┬───────┘
       ├──────────────────┬──────────────────┐
       ▼                  ▼                  ▼
┌─────────────┐   ┌─────────────┐   ┌──────────────┐
│ BiliCrawler  │   │ YTCrawler   │   │ IGCrawler    │
│ (HTTP API)   │   │ (Official   │   │ (Rod/Browser │
│              │   │  API)       │   │  Automation) │
└──────┬──────┘   └──────┬──────┘   └──────┬───────┘
       └──────────────────┴──────────────────┘
                          ▼
                   ┌─────────────┐
                   │  CSV 导出    │
                   └─────────────┘
```

定义统一的 `Crawler` 接口，每个平台各自实现，底层技术可以不同。既能利用 API 的高效，又能用浏览器自动化兜底。

## 3. "筛选博主"功能的可行性分析

### 3.1 问题

用户需要根据地区、博主类别、粉丝区间来筛选博主。但视频平台本身不提供此能力，关键字搜索用户也没有模糊搜索能力，结果严重失真。

### 3.2 唯一可行路径：自建博主数据库

#### 数据规模估算（以 YouTube 为例）

| 维度 | 数据量 |
|------|--------|
| YouTube 活跃频道总数 | ~5000万+ |
| 粉丝 > 1000 的频道 | ~1500万 |
| 粉丝 > 1万的频道（有营销价值） | ~300万 |
| 粉丝 > 10万的频道（中腰部博主） | ~30-50万 |

单条博主数据约 10KB，单平台（粉丝 > 1万）存储量约 **30 GB**。三个平台合计约 **50-100 GB**。

#### 爬取时间估算

| 方案 | 单条耗时 | 300万博主 | 50并发 |
|------|---------|----------|--------|
| 浏览器自动化 | 3-5秒 | ~139天 | ~3天（易触发反爬） |
| API/HTTP 请求 | 200-500ms | ~10天 | ~5小时 |
| YouTube Data API（免费配额） | 受限 | ~600天 | N/A |

#### 隐性代价

| 代价 | 说明 |
|------|------|
| 反爬对抗 | 大规模爬取需代理 IP 池，成本约 $50-200/月 |
| 数据时效性 | 博主数据动态变化，需定期全量更新 |
| 分类标注 | 平台不提供"博主类别"，需 NLP 分析自动分类 |
| 地区识别 | 博主地区信息不一定公开，需间接推断 |
| 法律风险 | 大规模爬取可能违反平台 ToS |

## 4. 分阶段实施计划

### 已确认：先实施阶段 0 + 阶段 1

```
阶段 0（MVP）：关键字搜索 + 视频元数据统计
├── 就是 youtube.py 已经在做的事情，用 Go 重写
├── 输入：关键字 → 输出：视频列表 CSV
├── 字段：标题、作者、播放次数、发布时间、视频时长、来源
└── 技术：按平台选择 API 直调 / 浏览器自动化

阶段 1：搜索结果中的博主聚合
├── 在阶段 0 的基础上，对搜索结果按作者聚合
├── 点进每个作者主页，爬取粉丝数、地区等信息
├── 输入：关键字 → 输出：博主列表（含粉丝数、视频数等）
└── 数据量小（一次搜索涉及的博主几十~几百个），无需建库

阶段 2（远期）：小规模博主库
├── 针对特定垂类（如"美妆"），爬取该类别下的博主
├── 建本地数据库（SQLite/PostgreSQL），支持筛选
└── 数据量：几千~几万条，可控

阶段 3（远期）：大规模博主库
├── 全品类爬取，代理 IP 池，定时更新
└── 需要考虑上述大规模代价
```

**阶段 1 是性价比最高的切入点**——不需要建库，但已经能实现"根据关键字找到相关博主，并按粉丝数筛选"的核心能力。
