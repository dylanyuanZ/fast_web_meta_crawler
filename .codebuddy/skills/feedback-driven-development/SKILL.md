# 反馈驱动开发方法（Feedback-Driven Development）

## 核心原则：快速迭代，用运行结果驱动修改

通过"修改→编译→运行→分析结果→修改"的紧密循环，让**程序的实际运行结果**指导每一步修改，而非凭推测编码。

**适用场景**：任何需要反复调试验证的编码任务（爬虫开发、解析器调试、并发模块优化、难以复现的 bug 修复等）。

---

## 强制流程

### Step 0: 初始化（进入循环前必做）

进入迭代循环前，**必须**完成以下初始化：

#### 0.1 权限与环境检查

| 检查项 | 操作 |
|--------|------|
| 进程管理权限 | 确认是否有 `kill`/`pkill` 权限，如无则向用户索要授权或准备替代方案（如写一个 kill 脚本） |
| 残留进程清理 | 检查并清理上次运行残留的进程（如 chromium、旧的 go 进程） |
| 残留数据清理 | 清理上次运行残留的临时文件（如 progress 文件、旧的输出文件） |
| 编译验证 | 确认当前代码可编译通过 |

**如果权限不足**：立即向用户说明需要哪些权限，等待授权后再进入循环。**禁止在循环中途才发现权限不足**。

#### 0.2 创建方案文档

根据当前反馈驱动的具体任务，在 `doc/design/` 下创建对应的任务目录，并在其中创建 `feedback_driven_report.md`：

```
doc/design/<task-name>/feedback_driven_report.md
```

例如：`doc/design/fix-412-blocking/feedback_driven_report.md`

初始内容包括：
- 问题描述
- 验证目标
- 运行命令

此文档将在整个反馈驱动开发过程中持续更新，是本次任务的**唯一方案文档**。

### Step 1: 明确验证目标

进入循环前，**必须**先明确本轮要验证什么：

| ✅ 好的验证目标 | ❌ 差的验证目标 |
|----------------|----------------|
| "验证 URL pattern 是否匹配所有请求" | "看看能不能跑通" |
| "验证翻页后数据是否与第一页不同" | "试试翻页" |
| "验证并发 10 个 goroutine 时是否有 race" | "跑一下并发" |

### Step 2: 执行迭代循环

```
修改代码 → 编译 → 运行测试 → 分析结果 → 更新文档 → 修改代码 → ...
```

**每次循环的执行规范：**

1. **修改代码**：只改与当前验证目标相关的最小改动
2. **编译**：确保编译通过，修复编译错误
3. **运行测试**：执行目标测试，收集完整输出
4. **分析结果**：
   - 测试通过 → 记录成功，进入下一个验证目标
   - 测试失败 → 从错误信息中提取**具体原因**，不要猜测
   - 输出不符合预期 → 对比"期望 vs 实际"，定位差异点
5. **更新文档**：将本轮的修改内容、运行结果、结果分析**立即**追加到 `feedback_driven_report.md`（详见「每轮循环必须更新文档」规则）
6. **修改代码**：基于分析结果做**针对性修改**，不做无关改动

### Step 3: 循环退出条件

当以下条件**全部满足**时退出循环：
- 所有验证目标均已通过
- 无编译警告/错误
- 测试输出符合预期

### Step 4: 结束与总结（退出循环后必做）

循环退出后，**必须**完成以下收尾工作：

#### 4.1 完善方案文档

将以下内容补充到 `feedback_driven_report.md`：

1. **修复总结表**：所有 Bug 的严重度、涉及文件、修复策略
2. **迭代效果对比表**：每轮迭代的关键指标变化
3. **关键优化思路**：本次开发中的核心技术决策和思路
4. **经验教训**：可复用的经验（如第三方库 API 验证、防御性超时等）

#### 4.2 清理

- 删除调试用的临时代码/文件
- 确认最终代码可编译通过
- 确认最终测试可通过

## 关键规则

### 单一变量原则

每次循环**只改一个东西**，这样才能从结果中准确归因：

| ❌ 一次改多处 | ✅ 一次改一处 |
|--------------|--------------|
| 同时改 URL pattern + 解析逻辑 + 翻页参数 | 先改 URL pattern → 验证 → 再改解析逻辑 |
| 同时加日志 + 改业务逻辑 | 先加日志观察 → 再改业务逻辑 |

**例外**：编译错误导致的连锁修改（如改了接口签名，必须同步改调用方）。

### 错误分析优先于修复

遇到失败时，**先完整分析错误**，再动手修改：

1. 读完整的错误信息和堆栈
2. 定位到具体的代码行
3. 理解**为什么**出错（不是**哪里**出错）
4. 然后再修改

### 每轮循环必须更新文档

每轮迭代循环结束后（无论成功还是失败），**必须**立即将以下内容追加到 `feedback_driven_report.md`：

| 必记内容 | 说明 |
|----------|------|
| 本轮修改了什么 | 代码改动的核心思路（1-2 句话） |
| 运行结果 | 成功/失败/超时，关键数据 |
| 结果分析 | 为什么成功/失败，下一步方向 |

**禁止**连续多轮迭代后才批量更新文档。文档是迭代循环的一部分，不是收尾工作。

### 保留调试痕迹

- 关键的调试日志在验证通过后再清理，不要边调边删
- 如果某个错误反复出现，记录到实测经验中

### 超时保护（防止无限等待）

程序卡住（hang）是迭代循环中最危险的情况——AI 会陷入无限等待，整个流程停滞。

**强制规则**：运行程序后，如果**超过 10 分钟没有新的日志输出**，必须：

1. **主动终止进程**（kill / Ctrl+C）
2. **将此视为超时错误**，记录为本轮循环的失败结果
3. **进入下一轮循环**，分析超时原因并做针对性修改

**超时分析方向**：

| 常见卡住原因 | 排查方式 |
|-------------|---------|
| 死锁（goroutine 互相等待） | 发送 SIGQUIT 获取 goroutine dump，或加 `pprof` |
| 网络请求无超时 | 检查 HTTP client / 浏览器操作是否设置了 timeout |
| 无限循环 | 检查循环退出条件是否可达 |
| 阻塞在 channel 操作 | 检查 channel 的发送/接收是否配对 |
| 等待外部资源 | 检查数据库连接、文件锁、外部服务响应 |

**预防措施**：在编写代码时，为所有可能阻塞的操作添加超时机制（`context.WithTimeout`、`time.After`、HTTP client timeout 等）。

### 上下文管理（防止上下文超限）

AI 的上下文窗口有限，长时间迭代会导致上下文超限，循环无法继续。

**强制规则**：

1. **控制日志读取量**：运行程序后，只读取日志的**最后 50 行**（`tail -50`），不要读取完整日志。如需更多信息，用 `grep` 过滤关键词（如 `ERROR`、`WARN`、`success`、`failed`）
2. **小规模测试时精简日志**：在首轮小规模测试中，如果发现程序输出大量重复/无效日志（如每个 HTTP 请求都打印完整 header），**立即修改代码减少日志量**，作为当轮循环的修复项之一
3. **每轮循环末尾沉淀到文档**：每轮循环结束后，将本轮的优化思路、修改内容、验证结果和结果分析追加到 `feedback_driven_report.md`
4. **跨轮次上下文重置**：如果上下文接近上限，告知用户"建议开启新会话继续"，新会话从 `feedback_driven_report.md` 中恢复上下文，无需重新阅读所有代码

**日志读取优先级**：

| 优先级 | 方式 | 示例 |
|--------|------|------|
| 1 | 只看退出码和最终统计 | `tail -5 output.log` |
| 2 | 过滤错误和警告 | `grep -E "ERROR\|WARN\|FAIL" output.log` |
| 3 | 看最后 N 行 | `tail -50 output.log` |
| 4 | 全量读取（仅在以上都不够时） | `cat output.log`（⚠️ 慎用） |

### 由小及大（渐进式测试范围）

刚写完的程序必然存在大量 bug，如果一上来就跑全量测试，既耗时又难以定位问题。

**强制规则**：新代码的首次验证，**必须先缩小测试范围**，跑通小案例后再逐步扩大到全量。

**缩小范围的常用手段**：

| 手段 | 示例 |
|------|------|
| limit 参数限制数量 | `--limit 3` 只处理前 3 个 author |
| 硬编码测试数据 | 用 1-2 条固定输入替代动态获取 |
| 跳过非核心分支 | 先只走主流程，忽略异常/边界处理 |
| 缩小并发度 | 先单协程跑通，再开并发 |

**渐进扩大节奏**：

```
小样本（1-3 条） → 中等样本（10-20 条） → 全量测试
```

每一级扩大前，**必须确认上一级已完全通过**。如果中等样本出现新问题，回到小样本级别修复后再扩大。

## 文档产物

一次完整的反馈驱动开发，产出**唯一一个**方案文档：

```
doc/design/<task-name>/feedback_driven_report.md
```

文档结构：

| 章节 | 内容 | 写入时机 |
|------|------|----------|
| 1. 背景 | 问题描述、验证目标、运行命令 | Step 0 初始化时 |
| 2. 问题分析 | Bug 根因分析 | 首轮循环分析后 |
| 3. 迭代过程 | 每轮的思路、修改、结果、分析 | **每轮循环结束后立即追加（强制）** |
| 4. 修复总结 | 总结表、效果对比、经验教训 | Step 4 结束时 |

## 与探针驱动开发的关系

本方法可**独立使用**，也可作为 `probe-driven-development`（探针驱动开发）的 Phase 3 衔接使用：

| 场景 | 入口 |
|------|------|
| 新平台爬虫开发 | 先走 `probe-driven-development`（探测→观察），再进入本方法 |
| 已有代码的调试/优化 | 直接使用本方法 |
| 修复已知 bug | 直接使用本方法 |
