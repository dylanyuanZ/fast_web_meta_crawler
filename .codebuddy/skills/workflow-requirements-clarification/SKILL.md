# 需求澄清

## 核心定位

**AI 负责调研代码背景，用户负责提供需求信息。**

### AI 职责划分

**AI 自己调研（读代码）**：
- 现有实现是怎样的
- 代码路径、数据流
- 已有的接口和数据结构
- 技术约束（如使用的框架）
- 相关模块的职责

**向用户询问**：
- 为什么需要这个功能
- 要解决什么问题
- 具体要做哪些功能
- 目标和成功标准
- 非目标（不做什么）
- 性能/兼容性等非功能需求

**规则**：如果信息可以从 codebase 获取，**AI 必须自己调研**，不问用户。

### 苏格拉底式提问

向用户询问需求信息时，采用**苏格拉底式提问**——刨根问底，理解问题本质：

| 技巧 | 说明 | 示例 |
|------|------|------|
| **追问 Why** | 不接受表面答案，追问到问题本质 | 用户："加个缓存" → AI："为什么需要缓存？慢到什么程度？目标是多少？" |
| **指出矛盾** | 发现用户表述中的矛盾时，引导用户澄清 | 用户："要高并发，但不能改接口" → AI："这两个目标是否冲突？" |
| **揭示遗漏** | 引导用户思考未考虑的场景 | AI："你考虑过 X 失败的情况吗？" |

### AI 生成内容的前置条件

1. **AI 已完成代码调研**（理解现状，并向用户确认）
2. **AI 已充分追问**（理解问题本质）

## 触发条件

- 用户描述功能需求但缺乏具体细节
- 用户请求开发新功能
- `spec.md` 不存在或需求章节为空

## 对话模式

### AI 角色边界

| AI 应该做 | AI 不应该做 |
|----------|------------|
| 提问，追问 Why | 未经请求就给答案 |
| 指出矛盾或遗漏 | 未追问清楚就生成内容 |
| 质疑不清晰的表述 | 跳过追问直接给建议 |
| **用户请求时**生成内容 | 生成后不询问用户意见 |

### 每轮对话结构

```
1. 说明当前阶段和目标
2. 提出一个开放式问题（让用户思考）
3. 等待用户回答
4. 评估用户回答：
   - 清晰完整 → 复述确认，进入下一问题
   - 模糊 → 追问具体含义
   - 有矛盾 → 指出矛盾
   - 有遗漏 → 引导思考
5. 用户澄清后，再次确认
```

## 工作流程

### Step 1: 创建 IDE 任务列表

```
todo_write(merge=false, todos=[
  {"id": "rc-1", "status": "in_progress", "content": "Step 1: 创建 IDE 任务列表"},
  {"id": "rc-2", "status": "pending", "content": "Step 2: [条件] 轻量提问（定位模块/功能域）"},
  {"id": "rc-3", "status": "pending", "content": "Step 3: 代码调研（AI 自主完成，范围确定后才执行）"},
  {"id": "rc-4", "status": "pending", "content": "Step 4: 澄清背景与问题"},
  {"id": "rc-5", "status": "pending", "content": "Step 5: 明确目标与边界"},
  {"id": "rc-6", "status": "pending", "content": "Step 6: 明确功能性需求"},
  {"id": "rc-7", "status": "pending", "content": "Step 7: 确认非功能性需求"},
  {"id": "rc-8", "status": "pending", "content": "Step 8: 评估复杂度（简单需求可删除 spec）"},
  {"id": "rc-9", "status": "pending", "content": "Step 9: 最终确认 spec.md"}
])
```

### Step 2: [条件] 轻量提问

> **目标**：确定需求涉及哪个模块/功能域，为后续代码调研提供方向。
>
> **触发条件**：AI 无法从用户的初始描述中判断出"涉及哪个模块/功能域"时，才进行轻量提问。
>
> - ✅ **跳过**："bilibili 爬虫的搜索接口加个排序参数" → 模块明确（crawler/bilibili），功能域明确（搜索）
> - ✅ **跳过**："csv 导出的列名不对" → 文件明确
> - ❌ **需要提问**："帮我优化一下性能" → 哪个模块？哪个场景？
> - ❌ **需要提问**："我们需要支持新的平台" → 哪个平台？

**轻量提问只问 1-2 个定位性问题**，不展开：
1. "涉及哪个模块？"
2. "大致要做什么（一句话）？"

**结束条件**：AI 能判断出涉及的模块/功能域。进入 Step 3。

### Step 3: 代码调研（AI 自主完成）

**目标**：理解现有实现，不询问用户

> **前置条件**：Step 2 已完成，AI 已确定需求的大致范围。

**AI 操作**：
1. 使用 `codebase_search` / `read_file` 等工具深度调研相关代码（模块结构、接口、依赖关系、数据流）
2. 自主发散探索上下游依赖、关联模块
3. 识别相关模块、接口、数据结构
4. 生成现状分析摘要

**向用户汇报（必须）**：
```
我先调研了一下相关代码：

**相关模块**：
- [文件路径1]: [职责说明]
- [文件路径2]: [职责说明]

**现有实现**：
[概述当前的实现方式、数据流、关键接口]

**技术约束**：
- [约束1，如使用的框架、协议]
- [约束2，如已有的接口规范]

请确认我的理解是否正确？有遗漏或错误的地方吗？
```

**结束条件**：用户确认理解正确。**必须等用户确认后才能进入 Step 4**。

### Step 4: 澄清背景与问题

**目标**：让用户说清楚"要解决什么问题"

**4.0 创建 spec.md（进入 Step 4 时立即执行）**：

```bash
mkdir -p doc/design/<feature>/
```

在 `doc/design/<feature>/spec.md` 创建 spec 文件（使用 spec 模板）。

**开场白**：
```
请描述一下：**现在遇到了什么问题？这个问题带来了什么影响？**
```

**追问方向**：
- "你说的 X 具体是指什么？"
- "影响范围有多大？有数据吗？"
- "没有这个功能，现在是怎么解决的？"

**结束条件**：用户能清晰回答"要解决什么问题"，AI 复述确认无误。

**实时更新 spec**：将确认内容写入 `1.1 问题描述`，同时将 Step 3 代码调研结果写入 `1.2 现状分析` + `1.3 主要使用场景`。

### Step 5: 明确目标与边界

**目标**：让用户定义"成功标准"和"不做什么"

**开场白**：
```
好的，问题我理解了。

接下来请你想一下：
- **这次的目标是什么？怎么算做完了？**
- **有什么是明确不做的？**
```

**追问方向**：
- "你说'性能要好'，具体是多少？怎么测量？"
- "这几个目标的优先级是什么？"
- "如果时间不够，哪些可以不做？"

**结束条件**：用户给出明确、可衡量的目标和边界。

**实时更新 spec**：将确认内容写入 `2. 目标` + `2.1 非目标`。

### Step 6: 明确功能性需求

**目标**：让用户说清楚"具体要做哪些功能来达成目标"

**开场白**：
```
目标明确了。

接下来请你描述一下：**具体需要哪些功能来达成这个目标？**
```

**追问方向**：
- "这个功能的输入是什么？输出是什么？"
- "有哪些操作/场景需要支持？"
- "用户/调用方会怎么使用这个功能？"

**结束条件**：用户给出具体的功能列表，AI 复述确认无误。

**实时更新 spec**：将确认内容写入 `3.1 功能性需求`。

### Step 7: 确认非功能性需求

**目标**：引导用户思考隐藏的约束条件

**开场白**：
```
最后想请你考虑一下：**有没有隐藏的约束条件？**

比如：性能要求、反爬策略、数据量限制？
```

**非功能性需求 checklist**（AI 根据需求场景判断是否相关）：

| 类别 | 何时询问 | 引导问题 |
|------|---------|---------|
| 性能 | 涉及大量请求、数据处理 | 并发量？速率限制？超时设置？ |
| 可靠性 | 涉及长时间运行的爬取任务 | 断点续爬？失败重试？ |
| 反爬 | 涉及新平台或大量请求 | 需要代理？Cookie 管理？频率控制？ |
| 数据质量 | 涉及数据解析 | 数据校验？缺失字段处理？ |
| 兼容性 | 涉及平台 API 变更 | API 版本？降级方案？ |
| 可扩展 | 涉及新平台接入 | 插件化？统一接口？ |

**结束条件**：用户确认关键约束，或明确表示"没有其他约束"。

**实时更新 spec**：将确认内容写入 `3.2 非功能性需求`。

### Step 8: 评估复杂度

**评估复杂度**：

| 复杂度 | 信号 | 处理方式 |
|--------|------|----------|
| **简单** | bug fix、配置调整、单点修改 | **删除 spec.md 及空目录**，直接进入 code-generation |
| **中等** | 涉及多文件、单模块功能 | 保留 spec，进入 Step 9 最终确认 |
| **复杂** | 跨模块、新特性、架构变更 | 保留 spec，进入 Step 9 最终确认 |

### Step 9: 最终确认 spec.md

**目标**：回顾 spec 前三章节的完整性，确认无遗漏。

**操作**：
1. 读取 spec.md 前三章节内容
2. 向用户展示摘要，确认无需补充或修改
3. 如有修改，用 `replace_in_file` 更新对应章节

**结束语**：
```
spec.md 前三章节已完成：doc/design/<feature>/spec.md
（已填写：1. 背景、2. 目标、3. 需求）

如果准备好了，说"开始设计"进入 system design 阶段。
```

---

## 强制规则

1. **spec.md 只填前三章节**：1. 背景、2. 目标、3. 需求。设计方案由 `workflow-system-design` 负责
2. **spec.md 路径**：`doc/design/<feature>/spec.md`
3. **简单需求删除 spec**：Step 8 评估为"简单"时，删除 spec 及空目录

## 反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 问用户"现有实现是怎样的" | AI 自己读代码调研（Step 3） |
| Step 3 只搜用户给出的模块，不探索上下游 | 以用户范围为起点，主动发散探索上下游依赖 |
| 一次问多个问题 | 每轮只问一个核心问题 |
| 所有澄清完成后才开始写 spec | Step 4 创建 spec，每步确认后实时写入 |
