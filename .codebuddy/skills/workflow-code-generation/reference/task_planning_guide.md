# 任务拆解指南

> 本指南用于将 spec.md 的设计方案拆解为结构化的实施任务清单（tasks.md）。
>
> **前置条件**：调用本指南前，SKILL.md 已确保 spec.md 存在且完整。本指南不负责检查 spec 的存在性或质量，只负责拆解。

## 拆解流程

### 第一步：深入理解 spec

1. 完整读取 spec.md
2. 识别所有设计决策和关键约束
3. 提取可交付产物清单（新文件、修改文件、删除文件）
4. 识别 spec 中明确提出的验收标准
5. **识别跨切面关注点**：错误处理、日志格式、配置项等贯穿多个任务的内容

### 第二步：代码调研

> 注：SKILL.md 步骤 1 已完成初步代码阅读。本步骤在此基础上**补充调研**任务拆解所需的细节。

**本步骤需要获取的信息**：
1. **修改面**：哪些函数/接口/struct 需要变更
2. **依赖面**：哪些调用方会受影响
3. **构建面**：哪些 go.mod / go.sum 需要更新
4. **每个预期任务的 context**：直接修改的文件 + 上游（谁调用/谁传入数据） + 下游（谁消费产出）

### 第三步：任务拆解

按以下原则拆解。当原则之间发生冲突时，按**优先级从高到低**裁决：

> **原则优先级**：可编译（原则 1） > 依赖拓扑（原则 3） > 风险前置（原则 6） > 粒度适中（原则 2）

#### 原则 1：任务有序、可渐进编译（最高优先级）

- 每个任务完成后，代码库必须能 `go build ./...` 通过（允许占位桩）
- 后续任务可以在前序任务的基础上渐进构建

**占位桩规范**：
```go
// Stub: will be implemented in Task N
func FooBar(req *Request) (*Response, error) {
    return nil, fmt.Errorf("FooBar: stub, see Task #N")
}
```

#### 原则 2：粒度适中

一个任务 = **一个内聚的变更单元**。

**必须拆分**：
- 任务涉及 **3+ 个独立设计决策** → 按决策边界拆
- 任务修改 **多个不相关包** → 按包拆

**必须合并**：
- 改动与相邻任务在**同一函数/同一文件**中，且单独拿出来没有独立验收价值
- 多处改动是**同一模式的机械重复**

#### 原则 3：依赖关系明确

- 每个任务标注前置依赖（`depends_on` 字段）
- 依赖必须是 DAG（无环）
- 在任务列表开头提供依赖关系总览图

#### 原则 4：每个任务可独立验收

每个任务必须有明确的验收标准：
- **编译验证**：`go build ./...` 通过
- **行为验证**：具体的功能检查点
- **不破坏约束**：`go vet ./...` 无新警告

| ✗ 不可操作 | ✓ 可操作 |
|-----------|---------|
| "确保接口正确" | "`go build ./...` 编译通过" |
| "功能正常" | "执行 `go test ./src/crawler/...` 全部 PASS" |

#### 原则 5：spec 全覆盖

- 每个 spec 的设计章节必须有对应的任务覆盖
- 在任务列表末尾提供 spec 章节 → 任务的映射表

#### 原则 6：安全变更序——先建后迁后删

1. **建**：创建新接口/实现（与旧代码并存）
2. **迁**：迁移调用方到新接口
3. **删**：删除旧代码

### 第四步：写入 tasks.md

使用 `write_to_file` 写入 `tasks.md`（与 spec.md 同目录）。**严格按以下模板生成**：

```markdown
# 实施任务清单

> 由 spec.md 生成
> 任务总数: N
> 核心原则: [一句话概括拆解策略]

## 依赖关系总览

Task 1 (描述)
  ↓
Task 2 (描述)  ← 依赖 Task 1

## 变更影响概览

### 文件变更清单

| 文件 | 操作 | 涉及任务 | 说明 |
|------|------|---------|------|
| `src/crawler/bilibili.go` | 修改 | Task 1, 3 | 新增搜索接口 |

## 任务列表

### 任务 1: [ ] [简短描述]
- 文件: `path/to/file.go`（新建/修改/删除）
- 依赖: 无
- spec 映射: spec 章节 X.Y.Z
- 说明: 详细说明这个任务要做什么
- context:
  - `path/to/file.go` — 直接修改目标
  - `path/to/caller.go:FuncA()` — 上游调用方
- 验收标准:
  - [ ] `go build ./...` 编译通过
  - [ ] [其他可操作的检查]
- 子任务:
  - [ ] 1.1: [具体步骤]
  - [ ] 1.2: [具体步骤]

## Spec 覆盖映射

| Spec 章节 | 任务 | 说明 |
|-----------|------|------|
| X.1 | Task 1, 2 | ... |
```

## 质量自检

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 1 | **依赖 DAG 无环** | 画出依赖图，确认无循环 |
| 2 | **每个任务编译可通过** | `go build ./...` 通过 |
| 3 | **每个任务有验收标准** | 验收标准可操作 |
| 4 | **spec 全覆盖** | 映射表无空白行 |
| 5 | **粒度合理** | 无单行任务，无跨 5+ 包任务 |
| 6 | **变更序安全** | 不存在"先删后建" |

## 硬性规则

1. **本阶段只生成任务清单**：输出 `tasks.md`，不修改其他任何文件
2. **必须先读代码**：不理解现有代码就拆任务是禁止的
3. **必须引用 spec**：每个任务的 `spec 映射` 字段必须有值
4. **必须可编译**：每个任务完成后 `go build ./...` 必须通过
5. **依赖必须 DAG**：如果发现循环依赖，重新拆分任务直到消除
6. **验收标准必须可操作**：不能是"确保正确"之类的模糊描述
7. **先建后迁后删**：禁止先删除再创建
8. **歧义必须记录**：spec 有歧义时必须在任务中标注假设
